# Final Report

## Project

**MoE-aware runtime scheduling enhancement for TensorRT-LLM**

Fixed project path:

- model: `Qwen/Qwen1.5-MoE-A2.7B-Chat`
- quantization: TensorRT-LLM `INT4 weight-only`
- hardware: `RTX 4060 Ti 16GB`

## Problem

MoE requests are not equally expensive at runtime. Routing skew can create expert or rank hotspots, which in turn amplifies:

- decode tail latency
- batch stragglers
- step latency variance

The default scheduler is good at capacity and general batching, but it does not explicitly treat MoE pressure as a first-class scheduling signal.

## Goal

Add the smallest possible runtime architecture enhancement that still feels like TRT-LLM work:

- not a new serving framework
- not a kernel project
- not an external router

Instead:

- introduce a minimal runtime resource model
- make scheduling consume an explicit `step plan`
- validate the effect on real MoE inference

### 5. Quantitative experiment path

This is the most important caveat in the whole project.

The internal patch exists and is version-aligned with TRT-LLM 1.2.1. However, on this machine the TRT-LLM PyTorch backend could not serve a matching quantized MoE path for final quantitative benchmarking. A GPTQ attempt failed with unsupported quantization config.

So the quantitative runs were executed on the real TensorRT engine backend by applying the same `resource_model -> step_plan` logic to batch composition in [scripts/run_patched.py](C:/26spring/nv项目/trtllm-moe-runtime-exp/scripts/run_patched.py).

That means:

- the architecture contribution is still real
- the model path is still real
- the benchmark path is still real
- but the final quantitative comparison is not a pure in-backend PyTorch patch benchmark

## Results

The complete tables are in:

### Balanced control

Patched path did not regress the balanced control:

- TTFT p90 improved from `0.0816s` to `0.0739s`
- E2E p90 improved from `1.4878s` to `1.4748s`
- throughput slightly increased from `278.55 tok/s` to `281.47 tok/s`

### Hot-Expert pilot

Patched path improved the hot-expert latency metrics:

- TTFT p90: `0.0748s -> 0.0112s`
- E2E p90: `1.8668s -> 1.5741s`
- TPOT p90: `0.0116s -> 0.0098s`

But it paid for that by shrinking effective batch size:

- throughput: `297.03 tok/s -> 99.57 tok/s`

### Hot-Rank pilot

The same pattern held:

- TTFT p90: `0.0737s -> 0.0131s`
- E2E p90: `1.9942s -> 1.7115s`
- step latency std also dropped materially
- throughput fell from `277.26 tok/s` to `99.37 tok/s`

### Hot-Expert full

The winning full compare kept the same direction on 24 requests:

- TTFT p90: `0.0733s -> 0.0111s`
- E2E p90: `1.8348s -> 1.5983s`
- TPOT p90: `0.0114s -> 0.0101s`
- throughput: `310.44 tok/s -> 99.81 tok/s`

## Interpretation

This v1 planner is best understood as a **latency-first isolation policy**.

It does exactly what it was designed to do:

- keep hot requests from stacking in the same batch
- reduce tail latency under MoE pressure
- remain safe on the balanced control

The obvious downside is that the current pressure budget is too conservative for throughput-sensitive serving. In the hot workloads it frequently collapses the batch to size 1.

## What the project demonstrates

Even with the backend limitation, this project still shows value for the TRT-LLM inference optimization track:

1. real TRT-LLM engine bring-up on constrained hardware
2. real understanding of TRT-LLM scheduler seams
3. explicit runtime resource modeling instead of heuristic-only patching
4. real MoE-specific workload design
5. honest tradeoff analysis, not a cherry-picked speedup claim

## Next iteration

If this were taken beyond the 24h slice, the next work item would be:

- adaptive pressure budgets that preserve more than one request per hot batch

That is the cleanest path to recover throughput while keeping the tail-latency benefit.
